<feature>
  <name>ThreadStore</name>
  <files>infra/scheduler/src/thread-store.ts, infra/scheduler/src/thread-store.test.ts</files>
  <behavior>
    ## Types

    - ThreadRecord: { convKey: string, channel: string, threadTs: string, createdAt: number, lastActivityAt: number }
    - ThreadMessage: { id: number, convKey: string, role: "user" | "assistant", content: string, slackTs: string | null, createdAt: number }

    ## Constructor(dbPath: string)

    - Opens SQLite database at dbPath (creates if not exists)
    - Creates parent directory if needed (mkdirSync recursive)
    - Enables WAL mode: `PRAGMA journal_mode = WAL`
    - Sets busy timeout: `PRAGMA busy_timeout = 5000`
    - Sets synchronous to NORMAL: `PRAGMA synchronous = NORMAL`
    - Creates `threads` table:
      - conv_key TEXT PRIMARY KEY (format: "channel:thread_ts")
      - channel TEXT NOT NULL
      - thread_ts TEXT NOT NULL
      - created_at INTEGER NOT NULL DEFAULT (unixepoch())
      - last_activity_at INTEGER NOT NULL DEFAULT (unixepoch())
    - Creates `thread_messages` table:
      - id INTEGER PRIMARY KEY AUTOINCREMENT
      - conv_key TEXT NOT NULL (FK to threads)
      - role TEXT NOT NULL CHECK (role IN ('user', 'assistant'))
      - content TEXT NOT NULL
      - slack_ts TEXT (nullable, for dedup)
      - created_at INTEGER NOT NULL DEFAULT (unixepoch())
    - Creates index on thread_messages(conv_key, created_at DESC)
    - Creates UNIQUE index on thread_messages(conv_key, slack_ts) WHERE slack_ts IS NOT NULL (for dedup)

    ## ensureThread(convKey: string, channel: string, threadTs: string): void

    - INSERT OR IGNORE into threads table
    - UPDATE last_activity_at = unixepoch() WHERE conv_key = convKey
    - Use a prepared statement for each, called in sequence

    ## addMessage(convKey: string, msg: { role: "user" | "assistant", content: string, slackTs?: string }): void

    - INSERT OR IGNORE into thread_messages
    - The OR IGNORE handles duplicate slack_ts (event retry dedup)
    - Also updates last_activity_at on the thread

    ## getMessages(convKey: string, opts?: { limit?: number }): ThreadMessage[]

    - SELECT from thread_messages WHERE conv_key = ? ORDER BY created_at ASC, id ASC LIMIT ?
    - Default limit: 20
    - Returns messages in chronological order (oldest first)
    - Note: query with ORDER BY created_at DESC, id DESC LIMIT ? then reverse, to get the LAST N messages in chronological order

    ## getThread(convKey: string): ThreadRecord | null

    - SELECT from threads WHERE conv_key = ?
    - Returns null if not found

    ## close(): void

    - Closes the database connection

<feature>
  <name>ConversationLock</name>
  <files>infra/scheduler/src/thread-mutex.ts, infra/scheduler/src/thread-mutex.test.ts</files>
  <behavior>
    ## ConversationLock class

    - Internal state: `locks: Map<string, Promise<void>>`

    ## acquire(key: string): Promise<() => void>

    - If key has an active lock, await it, then retry (loop until key is free)
    - Create a new Promise, store it in the Map keyed by `key`
    - Return a release function that:
      - Deletes the key from the Map
      - Resolves the stored Promise

    ## Pattern (from reference chat.ts)

    ```typescript
    class ConversationLock {
      private locks = new Map<string, Promise<void>>();

      async acquire(key: string): Promise<() => void> {
        while (this.locks.has(key)) {
          await this.locks.get(key);
        }
        let release!: () => void;
        const promise = new Promise<void>((resolve) => { release = resolve; });
        this.locks.set(key, promise);
        return () => {
          this.locks.delete(key);
          release();
        };
      }
    }
    ```

    This is a zero-dependency, promise-based per-key mutex. Directly adapted from the reference implementation.
  </behavior>
  <implementation>
    1. Create thread-mutex.ts with ConversationLock class
    2. Single class, single method, ~20 lines of implementation
    3. Console.log with `[thread-mutex]` prefix (only for debug/error, not every acquire/release)

    For testing:
    - Test: sequential acquire/release works
    - Test: second acquire on same key blocks until first releases
    - Test: different keys are independent (no cross-blocking)
    - Test: release + re-acquire on same key works (no deadlock)
    - Test: three concurrent acquires on same key serialize in order
  </implementation>
</feature>
