<feature>
  <name>SlackBridge</name>
  <files>infra/scheduler/src/slack-bridge.ts, infra/scheduler/src/slack-bridge.test.ts</files>
  <behavior>
    ## Types

    - SlackBridgeOptions: { botToken: string, appToken: string, repoDir: string, dbPath?: string }
    - Module-level state: let bot: SlackBot | null, let store: ThreadStore | null, let lock: ConversationLock

    ## startSlackBridge(opts: SlackBridgeOptions): Promise<void>

    1. Initialize ConversationLock (new instance)
    2. Initialize ThreadStore at opts.dbPath ?? path.join(opts.repoDir, ".youji", "threads.db")
       - Create .youji directory if it doesn't exist (mkdirSync recursive)
    3. Create SlackBot with:
       - botToken: opts.botToken
       - appToken: opts.appToken
       - onMessage: the message handler (see below)
       - onReaction: log reaction events (stub for Phase 3 approval handling)
    4. Call bot.start()
    5. Log "[slack-bridge] Started — listening for DMs"

    ## Message handler (onMessage callback)

    For each incoming SlackMessage:
    1. Derive convKey from message (channel + threadTs)
    2. Acquire ConversationLock for convKey
    3. Inside lock:
       a. ensureThread(convKey, message.channel, message.threadTs)
       b. addMessage(convKey, { role: "user", content: message.text, slackTs: message.ts })
       c. Load history: getMessages(convKey, { limit: 20 })
       d. Process message:
          - Phase 2 stub: format a simple acknowledgment response
          - e.g., `"Received your message. (Director intelligence coming in Phase 3)"`
          - In Phase 3, this is where Claude Agent SDK query will be injected
       e. Store response: addMessage(convKey, { role: "assistant", content: response })
       f. Reply via replyFn(response)
    4. Release ConversationLock in finally block (ensure release even on error)
    5. On error: log with [slack-bridge] prefix, reply with error message, release lock

    ## stopSlackBridge(): Promise<void>

    1. Call bot.stop() if bot exists
    2. Call store.close() if store exists
    3. Set module-level state to null
    4. Log "[slack-bridge] Stopped"

  </behavior>
  <implementation>
    1. Create slack-bridge.ts with module-level state for bot, store, lock
    2. The onMessage callback is the core pipeline: lock → store → process → store → reply → unlock
    3. Error handling: try/finally around the lock to ensure release
    4. The stub response includes message count from history for testability
       e.g., `"Got it. (${history.length} messages in this thread)"`
    5. Console.log with `[slack-bridge]` prefix

    For testing:

  </implementation>
